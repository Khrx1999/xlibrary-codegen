# xlibrary v0.2.0 — Implementation Spec

> **Status:** Design locked (grilled 2026-05-22). Ready to plan & implement.
> **Source of design decisions:** see Grill session 14 questions Q1–Q14.
> **Architectural records:** [`docs/adr/0001`](adr/0001-multi-language-emitter-scope.md),
> [`docs/adr/0002`](adr/0002-xlib-inline-comment-format.md).
> **Domain language:** see [`CONTEXT.md`](../CONTEXT.md).

## Scope

Four feature areas ship together (or in v0.2 → v0.5 if staged):

1. **Multi-language emitter** — adds `ts` + `python` targets to existing
   `robot` + `selenium`.
2. **Self-Healing Locators (proactive)** — capture top-3 alternatives per
   step + viewer badges.
3. **Re-record Step (full editing)** — `xlibrary patch` with
   replace/insert/delete/move/reorder across all four targets.
4. **Test Data Wizard** — extract literal values into variables via field
   context + value regex.

## 1. Multi-language emitter

### 1.1 CLI

```
xlibrary codegen [url] [options]

  -o, --output <file>          Output file (default: recorded.robot)
  -l, --lang <target>          robot | selenium | ts | python (default: inferred from -o)
  --save-actions [file]        Save the raw action stream as JSONL (default: <output>.jsonl)
  ... existing flags ...

xlibrary emit <actions.jsonl> [options]    [NEW]
  -l, --lang <target>          Required
  -o, --output <file>          Required
  --test-name <name>           Default: derived from JSONL metadata
```

### 1.2 Target inference (`-o` extension → `-l`)

| Extension              | Target     |
| ---------------------- | ---------- |
| `.robot`               | `robot`    |
| `.selenium.robot`      | `selenium` |
| `.spec.ts`, `.ts`      | `ts`       |
| `.py`                  | `python`   |
| _other / no extension_ | `robot`    |

Explicit `-l` always wins. Mismatched combo (`-l ts -o test.robot`) is allowed
but emits a warning.

### 1.3 Recorder mode matrix

| Target     | Direct mode (bundle-patch OK)      | JSONL bridge (fallback) |
| ---------- | ---------------------------------- | ----------------------- |
| `robot`    | Inspector writes `.robot` directly | We re-translate JSONL   |
| `selenium` | Inspector writes via our SLEmitter | We re-translate JSONL   |
| `ts`       | Inspector writes via Playwright JS | **❌ Hard fail**        |
| `python`   | Inspector writes via Playwright PY | **❌ Hard fail**        |

When user requests `ts`/`python` and direct mode is unavailable, fail with:

> TypeScript/Python output requires direct mode, but xlibrary's bundle patch
> for playwright-core@X.Y.Z is failing. Update xlibrary, downgrade
> playwright-core, or use `-l robot` instead.

### 1.4 Python flavor

Default `python` → `python-pytest` (industry standard). Plain `python` and
`python-async` are post-v0.2 if requested.

### 1.5 TS flavor

Default `ts` → `playwright-test`. Plain `javascript` is post-v0.2 if requested.

### 1.6 JSONL artifact format

Playwright's existing JSONL line schema. We add a single header line at top:

```jsonl
{
  "xlib": 1,
  "recorded-at": "2026-05-22T...",
  "browser": "chromium",
  "test-name": "Login Flow"
}
```

`xlibrary emit` reads the header to default `--test-name`, browser context,
etc.

## 2. Self-Healing Locators (proactive)

### 2.1 What we capture

For every step that has a selector, capture top-3 alternatives in addition to
the primary one Playwright already picks.

### 2.2 Where it lives in the output

Inline `xlib:` comment per [ADR-0002](adr/0002-xlib-inline-comment-format.md):

```robot
Click    role=button[name="Sign in"]    # xlib:step=5;alts=["data-testid=login","[aria-label='Sign in']"]
```

If the element has only one candidate (no alternatives), emit `alts=[]` is
omitted — only `xlib:step=N` is written.

### 2.3 Quality grade

Heuristic priority on the selector kind, plus a +1 bump if the selector
uniquely matches one element on the page:

| Kind                      | Base grade |
| ------------------------- | ---------- |
| `data-testid` / test-id   | A+         |
| `role` + accessible name  | A          |
| `label` text              | A          |
| `placeholder` text        | B          |
| visible `text` content    | B          |
| `css` (id / unique class) | C          |
| `xpath`                   | D          |

`+1 grade` when the candidate matches exactly 1 element on the page (e.g. a
plain `[type=email]` that's the only one).

### 2.4 Viewer badges

The viewer renders each step with the primary's grade as a colored chip, plus
the top-3 candidates listed below on hover/expand. **View-only in v0.2 — no
click-to-override.**

### 2.5 Source of candidates

Playwright's `generateSelector()` already returns
`{ selector, selectors: string[], elements }` — we need a new bundle-patch
(parallel to the existing 3) to ferry `selectors[]` through to the action
payload so our emitters can write it.

## 3. Re-record Step (full editing CLI)

### 3.1 CLI

```
xlibrary patch <file> [options]

  --at <id>                    Replace step (id = number or fuzzy content)
  --insert-after <id>          Record new steps to insert AFTER id
  --insert-before <id>         Record new steps to insert BEFORE id
  --delete <id>                Delete step
  --delete <from>-<to>         Delete range
  --move <from> to <to>        Reorder
  --range <from>-<to>          Replace range (combined with --at)

  --non-interactive            Fail-fast instead of pausing on replay failure
  --no-backup                  Skip the .bak file
```

### 3.2 Step identification

1. **Integer** (`--at 5`) — direct 1-indexed lookup by `xlib:step=5` marker.
2. **Fuzzy string** (`--at "Click Login"`) — case-insensitive substring match
   on the keyword line. Multi-match → print disambiguation:
   ```
   Step "Click" matches 3 steps:
     [3] Click    role=button[name="Login"]
     [7] Click    role=button[name="Submit"]
     [9] Click    role=link[name="Logout"]
   Re-run with --at 3 / --at 7 / --at 9
   ```

### 3.3 Replay flow

1. Read source file → extract action stream (via JSONL sidecar if present,
   else parse markers from source file).
2. Spin up replay engine; replay steps 1 to (target − 1).
3. **Failure handling:** if a step fails to replay → pause, prompt
   `[s]kip / [r]ecord from here / [a]bort`. With `--non-interactive`, abort.
4. Open recorder window at target step. User records new step(s).
5. On browser close, splice new steps into source position, re-write file.

### 3.4 Cross-language patching

Universal via `xlib:step=N` marker. Replace works for all four targets without
language-specific AST. New steps emitted by re-recording use the same language
as the source file.

### 3.5 Backup

Write `.bak` adjacent to source before any mutation. Opt out with
`--no-backup`. Single file only — no rolling versions.

## 4. Test Data Wizard

### 4.1 CLI

```
xlibrary codegen ... --extract-data         Post-record extraction prompt
xlibrary extract <file> [options]           Standalone retrofit on existing file
  -o, --output <file>                       Default: in-place edit (with .bak)
  --yes                                     Skip confirmation, apply all
```

### 4.2 Detection

Two-layer:

1. **Field context** (priority — uses selector semantics):
   | Selector signal | Variable name |
   | -------------------------------- | ------------------------------ |
   | `[type=email]` | `VALID_EMAIL` |
   | `[type=password]` | `VALID_PASSWORD` |
   | `[type=tel]` | `VALID_PHONE` |
   | `[autocomplete=username]` | `USERNAME` |
   | `[autocomplete=current-password]`| `CURRENT_PASSWORD` |
   | `[name=email]` / similar | `EMAIL` |

2. **Value regex** (fallback when no field context):
   - URL pattern → `BASE_URL` or `URL_<n>`
   - Email shape → `EMAIL_<n>`

### 4.3 Dedup

Same value used in multiple steps → single variable, all sites reference it.
Different values for the same semantic category → numbered suffix
(`VALID_EMAIL`, `VALID_EMAIL_2`).

### 4.4 Output

Variables go into the test's own file:

- **Robot:** `*** Variables ***` section above `*** Test Cases ***`.
- **TS:** `const VALID_EMAIL = '…';` at top of file (outside `test()`).
- **Python (pytest):** module-level constants above the test functions.
- **Selenium (.robot):** same as Robot.

### 4.5 UX

Default flow: print a diff preview, prompt to confirm:

```
xlibrary extract test.robot
─── Detected 3 variables ──────────────
- ${VALID_EMAIL}       = "qa@example.com"   (2 sites)
- ${VALID_PASSWORD}    = "Hunter2!"         (1 site)
- ${BASE_URL}          = "https://app.com"  (1 site)
─── Diff preview ──────────────────────
  *** Variables ***
+ ${VALID_EMAIL}       qa@example.com
+ ${VALID_PASSWORD}    Hunter2!
+ ${BASE_URL}          https://app.com
...
Apply? [Y/n]
```

`--yes` skips the prompt for scripting.

## Cross-feature integration

- **Multi-lang × Self-Healing** — `xlib:alts` payload works in every target's
  comment syntax. No language-specific divergence.
- **Multi-lang × Patch** — `xlib:step=N` marker is universal. Patch operates
  on any target.
- **Multi-lang × Wizard** — extraction emits language-appropriate variable
  syntax; detection logic is language-independent (operates on action stream).
- **Self-Healing × Patch** — re-recorded steps get freshly-computed `alts`.
- **Self-Healing × Wizard** — independent; alts describe locators, vars
  describe values.
- **Patch × Wizard** — independent; both operate on the same `xlib:step=N`
  marker substrate, so they compose.

## Release sequencing (if staging is preferred)

Reality check: shipping all four together is ~4–8 weeks of focused work.
Suggested staging if the user prefers smaller, faster releases:

| Version | Ships                                                        |
| ------- | ------------------------------------------------------------ |
| v0.2.0  | Multi-language + JSONL artifact + `emit` subcommand          |
| v0.2.x  | Self-Healing (file + viewer badges)                          |
| v0.3.0  | Re-record Step (full editing CLI)                            |
| v0.3.x  | Test Data Wizard                                             |
| v0.4.0+ | Runtime healing, viewer step-editor UI, click-to-override, … |

This ordering puts the highest-value, lowest-risk pieces first and defers the
patcher (largest surface) until after the file-format conventions are stable.

## Open implementation questions (deferred to planning phase)

These weren't grilled in detail — they'll need to be resolved during the
implementation plan:

1. **Bundle-patch #4** — exact regex / hook point to ferry `selectors[]` from
   the recorder injected script through to our emitter input.
2. **Naming-convention conversion** — `--test-name "Login Flow"` →
   `def test_login_flow(page):` (snake_case) vs `test('Login Flow', …)`
   (kept as string) vs `Login Flow` (RF). Per-language formatter responsibility.
3. **Variable name collision** — `${BASE_URL}` already defined in source vs
   detected by wizard. Skip / rename / abort.
4. **Action stream version compatibility** — `xlibrary emit` reading a JSONL
   from an older xlibrary. Header version field decides; how to migrate.
5. **Viewer state for multi-feature interaction** — badges + patch markers +
   replay state coexist; UI design needs a unified pass.
